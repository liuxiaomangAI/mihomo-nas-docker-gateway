# Mihomo 在飞牛 NAS 上的部署说明

更新时间：2026-03-30

## 目标

这套配置用于在 NAS 上部署 Mihomo，实现以下目标：

- 全局透明代理（TUN）
- DNS 防污染
- 订阅节点自动更新
- GEO 数据自动更新
- 配置可迁移、可恢复、可长期运行

## 网络拓扑

```
主路由 -> NAS（Mihomo）-> 局域网设备
```

示例 NAS IP：

```text
192.168.5.21
```

请替换为你自己的 NAS 局域网 IP。

## 关键原则

部署是否稳定，主要取决于下面四点：

1. 预先放入本地 Geo 文件，避免首次启动时下载失败导致卡死。
2. 配置 `proxy-server-nameserver`，避免代理节点域名解析死锁。
3. 使用 `proxy-providers`，让订阅节点可以自动更新。
4. 启用 `geo-auto-update`，保证 GEO 数据可长期维护。

## 部署前准备

### 1. 准备 4 个 Geo 文件

首次启动前，需要先把下面 4 个文件手动放入 Mihomo 配置目录。

- `GeoSite.dat`
  下载地址：[MetaCubeX geosite.dat](https://github.com/MetaCubeX/meta-rules-dat/releases/download/latest/geosite.dat)
- `GeoIP.dat`
  下载地址：[MetaCubeX geoip-lite.dat](https://github.com/MetaCubeX/meta-rules-dat/releases/download/latest/geoip-lite.dat)
- `country-lite.mmdb`
  下载地址：[MetaCubeX country-lite.mmdb](https://github.com/MetaCubeX/meta-rules-dat/releases/download/latest/country-lite.mmdb)
- `GeoLite2-ASN.mmdb`
  下载地址：[MetaCubeX GeoLite2-ASN.mmdb](https://github.com/MetaCubeX/meta-rules-dat/releases/download/latest/GeoLite2-ASN.mmdb)

原因很简单：NAS 在还没连通代理前，通常无法直接从 GitHub 拉取这些文件。如果首次启动时依赖在线下载，常见结果是：

- 启动卡住
- Geo 下载失败
- `context deadline exceeded`

### 2. 准备一个可用订阅链接

你需要准备一个 Clash / Mihomo 兼容的订阅链接，要求如下：

- 至少包含可用代理节点
- 支持 Mihomo 可识别的订阅格式
- 能长期稳定更新

将该链接填入 `config.yaml` 中 `proxy-providers.lazydoge.url` 的位置。

推荐一个我已经使用了5年的（非常稳定，推荐使用旗舰日本节点）：https://www.xcjs123.com/auth/register?code=EsTMyvDQhJMjC5Lp

## 目录结构

NAS 上建议使用类似下面的目录结构：

```text
/vol1/1000/Docker/mihomo/
├── config.yaml
├── docker-compose.yml
├── GeoSite.dat
├── GeoIP.dat
├── country-lite.mmdb
├── GeoLite2-ASN.mmdb
├── cache.db
├── providers/
│   └── lazydoge.yaml
└── backup/
```

说明：

- `config.yaml`：Mihomo 主配置文件
- `docker-compose.yml`：容器启动配置
- `providers/lazydoge.yaml`：订阅拉取后自动生成
- `cache.db`：运行后自动生成
- `backup/`：建议预留，用于备份配置

## config.yaml 说明

当前仓库内已经包含可直接使用的 [config.yaml](./config.yaml)。

使用前请至少修改以下内容：

- 将订阅地址替换为你自己的链接
- 按需调整节点分组名称与筛选规则
- 将文中的 NAS IP 替换为你自己的局域网地址

核心配置点如下：

- `tun.enable: true`
  用于启用透明代理。
- `dns.enhanced-mode: fake-ip`
  用于实现稳定的 DNS 分流。
- `proxy-providers`
  用于自动拉取并更新订阅节点。
- `geo-auto-update: true`
  用于自动更新 GEO 数据。
- `proxy-server-nameserver`
  用于专门解析代理节点域名，这是稳定启动的关键之一。

如果没有 `proxy-server-nameserver`，常见问题包括：

- 节点域名解析失败
- fallback DNS 超时
- 代理链路无法建立

## docker-compose.yml

仓库内的 [docker-compose.yml](./docker-compose.yml) 内容如下：

```yaml
services:
  mihomo:
    image: metacubex/mihomo:latest
    container_name: mihomo
    restart: unless-stopped
    network_mode: host
    privileged: true
    cap_add:
      - NET_ADMIN
    volumes:
      - /vol1/1000/Docker/mihomo:/root/.config/mihomo
```

请确认宿主机挂载路径与你 NAS 中的实际目录一致。

## 在飞牛 NAS 中部署

### 1. 上传文件

将以下文件放到 NAS 目录中：

- `config.yaml`
- `docker-compose.yml`
- `GeoSite.dat`
- `GeoIP.dat`
- `country-lite.mmdb`
- `GeoLite2-ASN.mmdb`

### 2. 使用 Docker Compose 创建项目

在飞牛 NAS 的 Docker 管理界面中：

1. 进入 `Compose -> 新增项目`
2. 项目名称填写 `mihomo`
3. 项目路径选择：

```text
/vol1/1000/Docker/mihomo
```

4. 如果系统检测到已有 `docker-compose.yml`，直接选择使用现有配置创建项目
5. 确认配置无误后执行部署

### 3. 判断是否部署成功

部署成功后应满足：

- 容器 `mihomo` 处于运行状态
- 没有持续报错或反复重启

如果启动失败，优先检查：

- 挂载路径是否正确
- `docker-compose.yml` 是否存在
- `config.yaml` 是否填写完整
- 4 个 Geo 文件是否已经放到位

## 控制面板访问

Mihomo 外部控制器默认暴露在：

```text
http://192.168.5.21:9090
```

请把上面的 IP 换成你的 NAS 地址。

如果只想使用图形界面管理节点，可以用在线面板：

- [metacubexd](https://metacubex.github.io/metacubexd/)

在面板中填入：

```text
API Base URL: http://你的NASIP:9090
```

## 验证部署是否正常

### 1. 测试外网连通

```bash
curl -I https://www.google.com
```

正常情况下应返回 `200` 或可访问响应头。

### 2. 测试出口 IP

```bash
curl ifconfig.me
```

输出应为当前所选代理节点对应的出口 IP。

### 3. 查看运行日志

```bash
docker logs mihomo
```

你应该重点确认：

- 国内域名走 `DIRECT`
- OpenAI / Telegram 等规则命中代理分组
- 没有持续的 GEO 下载失败或 DNS 解析报错

### 4. 查看 GEO 相关日志

```bash
docker logs mihomo | grep GEO
```

如果在 NAS 中没有 `grep`，就直接查看完整日志。

## 局域网其他设备接入

如果你希望其他设备通过 NAS 实现“无感代理”，可以把这些设备的网关和 DNS 指向 NAS：

```text
网关：192.168.5.21
DNS：192.168.5.21
```

实际填写时请替换为你的 NAS IP。

是否适合这样做，取决于你的网络结构和主路由能力。若你的主路由支持 DHCP 自定义网关或 DNS，可以在主路由统一下发。

## 自动更新机制

本方案包含两类自动更新：

- 节点更新：由 `proxy-providers` 按订阅周期自动拉取
- GEO 更新：由 `geo-auto-update` 定时更新

## 长期稳定运行的关键

下面两项建议始终保留：

- 本地 Geo 文件
- `proxy-server-nameserver`

这两项直接影响首次启动成功率与后续稳定性。
